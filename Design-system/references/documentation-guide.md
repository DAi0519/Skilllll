# 设计系统说明文档指南 (Documentation Guide)

## 一、文档的必要性
- 设计系统完成后**必须**建立说明文档，是项目成败的关键
- 文档要让**毫无设计经验的人**也能正确使用
- 需要设计师和开发人员**共同完成**

---

## 二、文档整体结构

### 推荐目录

```
设计系统文档
├── 1. 概述 (Overview)
│   ├── 设计原则 (Design Principles)
│   ├── 快速开始 (Getting Started)
│   └── 更新日志 (Changelog)
├── 2. 基础 (Foundations)
│   ├── 颜色 (Color)
│   ├── 字体 (Typography)
│   ├── 间距 (Spacing)
│   ├── 网格 (Grid)
│   ├── 图标 (Iconography)
│   ├── 高度/阴影 (Elevation)
│   └── 动效 (Motion)
├── 3. 组件 (Components)
│   ├── 按钮 (Button)
│   ├── 输入框 (Input)
│   ├── ... (每个组件独立页面)
│   └── 表格 (Table)
├── 4. 模式 (Patterns)
│   ├── 表单 (Forms)
│   ├── 导航 (Navigation)
│   └── 数据展示 (Data Display)
├── 5. 指南 (Guidelines)
│   ├── 无障碍 (Accessibility)
│   ├── 响应式设计 (Responsive)
│   ├── UX 文案 (UX Writing)
│   └── 国际化 (i18n)
└── 6. 资源 (Resources)
    ├── 预览页 / 示例页面
    ├── 开发工具
    └── Design Token 参考
```

## 三、基础（Foundations）文档规范

每个基础模块应包含:

| 章节 | 内容 |
|------|------|
| 概述 | 该模块是什么、为什么重要 |
| 使用规范 | 如何正确使用 |
| Token 列表 | 所有相关 token 及其值 |
| Do & Don't | 正确和错误示例（图文对比） |
| 无障碍说明 | 相关的无障碍要求 |
| 最佳实践 | 实际使用案例 |

## 四、组件文档规范 (13项标准)

每个组件页面应包含以下内容:

### 1. 概述/描述 (Overview)
- 组件是什么
- 什么时候使用

### 2. 何时使用/何时不使用 (When to use / When not to use)
- 使用场景判断指南
- 与相似组件的区分

### 3. 解剖图 (Anatomy)
- 标注组件各部分的可视化图示
- 每个部分的名称和作用

### 4. 变体 (Variants)
- 所有变体的视觉展示
- 每种变体的使用场景

### 5. 尺寸 (Sizes)
- 可用的尺寸选项
- 各尺寸的适用场景

### 6. 状态 (States)
- 所有交互状态的视觉展示
- Default, Hover, Active, Focus, Disabled, Error, Loading

### 7. 行为 (Behavior)
- 交互模式
- 动画效果
- 响应式行为

### 8. 内容指南 (Content Guidelines)
- 文本长度限制
- 语气和措辞
- 大小写规范

### 9. 无障碍 (Accessibility)
- ARIA 角色和属性
- 键盘交互表格
- 屏幕阅读器行为

### 10. API/属性表 (Props Table)
- 每个属性的类型、默认值、描述

### 11. 交互示例 (Live Examples)
- 可交互的代码示例
- Playground 环境

### 12. Do & Don't
- **正确** 使用的视觉示例
- **错误** 使用的视觉示例
- 每个示例配简短说明

### 13. 相关组件 (Related Components)
- 相似或互补的组件链接

## 五、文档编写原则

### 1. 包容性
- 文档要让毫无设计经验的人也能正确使用
- 对专业术语进行通俗解释
  - 色相 (Hue): 颜色的种类
  - 饱和度 (Saturation): 颜色的鲜艳程度
  - 亮度 (Lightness): 颜色的明暗程度

### 2. 视觉化
- 使用**图文搭配**方式呈现说明
- 提供实际使用案例帮助理解
- Do & Don't 必须配图

### 3. 标准化
- 对专业术语进行通俗解释
- 需要专业**文案人员**统一润色

### 4. 详细规范
- 每个操作步骤都要明确说明
- 对可能产生风险的操作要特别标注（如破坏性操作）
- 文案简洁，避免复杂逻辑词语

## 六、文档存放方式

### 1. Markdown 文档
- 使用仓库内 Markdown 持续维护设计系统规范
- 优势: 版本可追踪、易于和代码同步
- 劣势: 交互展示能力有限

### 2. 独立网站
- 建立专门的设计系统网站
- 推荐工具: Storybook, Docusaurus, VitePress, Nextra
- 优势: 开发设计共同使用，可嵌入交互示例

### 3. 第三方平台
- Notion、语雀、Confluence 等知识库
- 优势: 提供标准化文档框架，方便团队共同维护

### 4. 选择建议
| 团队规模 | 推荐方案 |
|---------|---------|
| 个人/小团队 | Markdown + Notion/语雀 |
| 中型团队 | Storybook + 文档站 |
| 大型企业 | 专业文档平台 + 定制网站 |

## 七、文档迭代

### 流程
```
初稿 → 用户测试、收集反馈 → 优化 → 发布 → 持续更新
```

### 原则
- 收集其他设计师和开发人员的反馈
- 文档需要随设计系统同步更新
- 根据反馈不断改进文档内容
- 维护更新日志 (Changelog)
- 标注 Breaking Changes
