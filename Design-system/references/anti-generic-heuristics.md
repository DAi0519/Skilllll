# 反泛化启发式与视觉杠杆词库 (Anti-Generic Heuristics)

这个文件用于解决一个高频问题：

很多设计系统方案看起来“完整”，但仍然像默认 AI 产物。

常见原因不是 token 不够，而是：

- 没有明确的视觉杠杆
- 没有写出禁止项
- 在关键地方退回了默认 SaaS 审美

这个参考文件的目标是：

1. 帮你快速提炼 2-4 个真正决定气质的视觉杠杆
2. 帮你识别常见 generic 失败模式
3. 帮你把“更有设计感”转成可执行的设计动作

---

## 一、什么时候一定要读这个文件

这些场景默认先读本文件：

- 用户说“高级一点”“更有设计感”“不要那么像 AI”
- 用户要求参考竞品气质，但不想直接像对方
- 你自己已经写出一版方案，但感觉像“标准 SaaS 模板”
- 页面出现了默认渐变、大圆角卡片、发灰阴影、紫色主色、玻璃拟态等高频套壳特征

---

## 二、先找 2-4 个视觉杠杆

不要从“我要做得很高级”开始，要从“这次最关键的 2-4 个视觉杠杆是什么”开始。

### 视觉杠杆的定义

视觉杠杆是指：一旦改变，整个气质会立刻改变的设计变量。

好的视觉杠杆通常来自这些维度：

- 画布与表面
- 排版
- 色彩策略
- 圆角与边框
- 阴影与深度
- 信息密度
- 媒体使用方式
- 交互语气

### 常用视觉杠杆词库

#### 1. 画布与表面

- 纯黑影院画布
- 暖白纸感背景
- 冷白展厅画布
- 单色 chrome
- 表面分层很浅
- 表面分层很深
- 内容即颜色源
- 全幅媒体驱动

#### 2. 排版

- 轻字重大标题
- 极紧字距标题
- 巨型 display 作为结构元素
- mono kicker / mono 标签系统
- serif editorial 标题
- sans + mono 工程化组合
- 标题靠 scale，不靠字重
- 正文保持隐身

#### 3. 色彩

- 单一强调色
- 黑白 chrome + 品牌强调色
- 暖中性色体系
- 冷中性色体系
- 高饱和内容色，低存在 UI 色
- 语义色极度克制
- 品牌色只用于 CTA 和 active
- 色彩只出现在媒体里

#### 4. 圆角与边框

- 低圆角保守组件
- 完全直角编辑感
- pill / circle 几何优先
- 边框重于阴影
- 阴影重于边框
- shadow-as-border
- hairline 分隔体系
- 细描边容器

#### 5. 深度与材质

- 品牌化阴影
- 近乎无阴影
- 层叠阴影系统
- vignette 文本保护层
- 禁用 glassmorphism
- 只允许局部 blur
- 深色 inset 阴影
- 只用边框建立层级

#### 6. 布局与密度

- 超宽留白
- 新闻纸高密度
- 工程化网格
- 单列叙事流
- 卡片矩阵
- 仪表盘密集布局
- 慢节奏 section rhythm
- 内容压缩 chrome 让位

## 三、默认 AI 失败模式

### 1. 渐变滥用

现象：

- hero 背景默认上渐变
- CTA 也渐变
- 卡片也渐变
- 强调色没有主次

为什么 generic：

- 这是模型最常见的“显得高级”的偷懒方式
- 没有真正建立层级，只是在增加视觉噪声

改法：

- 默认先禁止渐变
- 只有当品牌真的依赖 gradient language 时才启用
- 一旦启用，限定只出现在 hero 或单一品牌区域，不得全站扩散

### 2. 大圆角 SaaS 卡片化

现象：

- 所有卡片、按钮、输入框都变成 16px-24px 圆角
- 页面出现一堆柔软胶囊块

为什么 generic：

- 这是最常见的 AI-SaaS 模板指纹

改法：

- 先判断风格究竟是直角、低圆角还是 pill geometry
- 如果不是明确走“友好圆润”路线，优先压到 0/4/6/8px

### 3. 无方向的中性阴影

现象：

- 阴影发灰
- 模糊一片
- 所有层级都差不多

为什么 generic：

- 它不能表达品牌，只是在机械模拟“浮起来”

改法：

- 明确是“边框优先”还是“阴影优先”
- 如果用阴影，写清楚色相、层级、用途
- 不要默认 `rgba(0,0,0,0.12) 0 8px 24px`

### 4. 多强调色竞争

现象：

- CTA 一个颜色
- 标签一个颜色
- 图标又一个颜色
- hover 再来一个颜色

为什么 generic：

- 模型在试图“做丰富”，但会让层级失焦

改法：

- 默认一个主强调色
- 语义色只在状态场景出现
- 装饰色不进入主交互层

### 5. 抽象词没有证据

现象：

- 文档写“高级、未来感、现代”
- 但没有任何 token 或组件决策支撑它

为什么 generic：

- 这类词不会真正约束生成

改法：

- 每个抽象词后面都补视觉证据
- 例如“工程化 = 极紧字距 + mono 标签 + 单色 UI + 边框重于阴影”

### 6. 默认玻璃拟态

现象：

- 背景模糊
- 半透明面板
- 反光边框
- 到处都是 frosted card

为什么 generic：

- 这是另一个高频“看起来复杂”的伪进阶套路

改法：

- 默认禁用 glassmorphism
- 只有当产品语义真的需要材质感时，才允许局部使用

## 四、把“更有设计感”翻译成动作

用户常说的需求，应该这样转译：

### “高级一点”

优先考虑：

- 更克制的色彩数量
- 更明确的字体层级
- 更少但更准的材质变化
- 更高质量的边框/阴影系统

不要默认：

- 渐变
- 发光
- 大圆角

### “更像大厂”

优先考虑：

- 更稳定的组件几何
- 更清晰的状态层级
- 更可复用的 token 角色
- 更严格的禁止项

不要默认：

- 复制某家品牌色
- 复制某家专有字体

### “更有设计感”

优先考虑：

- 让页面拥有一个真正鲜明的主视觉矛盾
- 例如：极简 UI + 高饱和内容，或冷白版心 + 暖文字，或超大标题 + 极细边框

不要默认：

- 让每个元素都更花

## 五、产出前的反泛化检查

在输出方案前，快速自检：

- 我是否已经写出 2-4 个视觉杠杆？
- 我是否写出了 3-6 条 Anti-generic Rules？
- 这个方案能否一眼和“默认 SaaS 模板”区分开？
- 我是否明确了边框、阴影、圆角、强调色的优先级？
- 我是否禁止了最可能跑偏的那几种默认设计习惯？

如果有任意一项答不上来，先不要生成 token。

## 六、推荐输出格式

在 Step 1 的方案阶段，建议显式写出这两段：

```markdown
### Visual Levers
- ...
- ...
- ...

### Anti-generic Rules
- ...
- ...
- ...
```

后续 `design-system.md` 中，至少要把这些内容落到：

- `Visual Theme & Atmosphere`
- `Do's and Don'ts`
- `Agent Prompt Guide`

---

## 七、不要把“反泛化”理解成“更花”

真正的 anti-generic 往往意味着：

- 更少颜色，而不是更多颜色
- 更少组件姿态，而不是更多组件姿态
- 更明确的主次，而不是每个地方都想出彩
- 更强的约束，而不是更自由

所以目标不是“更炫”，而是“更有识别度、更有纪律”。
