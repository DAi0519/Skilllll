# AI 产品经理工作流程相关

这份文档基于截图内容整理，尽量保留原文结构、标题和表述，不做过度改写。

## 8、AI 产品经理工作流程相关

传统 PRD 关注：

- 页面
- 功能点
- 接口逻辑

AI 产品新增关注：

1. 模型行为不可控（概率性）
2. 输入输出不确定（Prompt、推理路径）
3. 决策链条不同（不是按按钮，而是执行任务）
4. 需要长期优化，而不是开发完就结束

## 8.1 PRD 的 9 大核心组成部分

PRD（Product Requirement Document）是产品经理的核心产出，阐述了用户诉求应该用什么样的方式来实现。一份 PRD 通常至少包括 9 块内容。

注：红字是 AI 产品 PRD 新增部分。

1. 需求背景与目标（是什么问题、为什么要做、产品目标可量化）
2. 现有流程梳理（当前是怎么做的、用户故事）
   + Agent 故事（Agent Story，将用户故事转化为 Agent 故事）
3. 用户旅程设计（将来用户怎么使用你的产品）
   + Agent 流程（Agent Workflow，将用户旅程转化为 Agent 流程）
4. 原型图或界面草图（产品长什么样）
5. 产品功能框架（要做哪些模块和能力）
6. 详细功能点：这些功能的细节
7. 提示词设计
8. 评估指标与评测数据集
9. 非功能性需求（埋点、性能、安全等）

## 8.2 AI 产品经理的 PRD vs 传统产品经理的 PRD

| PRD 模块 | AI 产品经理 | 传统产品经理 | 是否一致 |
| --- | --- | --- | --- |
| 需求分析、产品定位、用户故事 | -- | -- | 一致 |
| 用户业务流程 | 涉及到大模型/小模型工作流（比如 OCR、大模型语义理解要素抽取、RAG 等） | -- | 略不一致 |
| 产品功能设计及功能清单 | 问答类产品功能非常简化；增加大模型/Agent 功能节点 | -- | 略不一致 |
| 详细功能点 | 操作交互 | -- | 略不一致 |
| 提示词 | 涉及工作流各节点的提示词 | -- | 不一致 |
| 原型图设计 | 1. 问答类产品功能简化；2. 文档撰写场景通过问答交互方式会和传统方式不一样；需要用户确认、反馈的时候，会在对话框里呈现出 GUI（图形操作页面）来实现 | -- | 略不一致 |
| 数据评测集 | 评估模型效果 | -- | 不一致 |
| 评估体系 | 评估模型效果 | -- | 不一致 |

说明：

PRD 模板没有用。PRD 的核心是表达清楚需求，不是写八股文。你的 PRD 有以上几个模块一般就够用了（有些会涉及到性能需求等），关键的是里面具体的内容细节，那项目之间差异很大，也不存在套用模板。

具体会发生如下变化。

## 8.2.1 将用户故事转化为 Agent 故事

### 1. 对产品经理

明确 AI 在每个任务中的输入、能力和输出；思考 AI 的边界和依赖的工具；为评估技术可行性提供依据。

### 2. 对开发团队

明白 AI 需要哪些 State 数据；清楚要准备哪些工具（API/函数）；理解输出格式要求；提示词如何设计。

### 3. 对测试团队

能根据 Agent 故事构建测试用例；明确输出格式如何校验；知道该准备哪些上下文数据。

### 输出【用户故事】

传统 PRD 的写法：

- 作为【用户角色】
- 在【某使用场景】下
- 为了【达成某目标】
- 需要【获得某能力或支持】

示例：电商 AI 助手

作为一名电商平台的普通消费者，
在遇到订单状态不清楚、物流信息更新不及时、售后流程复杂的场景下，
为了快速了解订单情况并获得准确的售后指引，
需要一个能自动解答问题、并可根据我的订单信息提供个性化回复的智能客服助手。

### Agent 故事结构写法

- 作为【Agent 角色】
- 在【某任务场景】下
- 为了【完成某个目标】
- 需要【上下文信息 / 工具 / 能力支持】

示例：电商 AI 助手 - 订单查询 Agent

作为【订单查询 Agent（OrderInquiryAgent）】，
在【用户询问订单状态/物流的任务场景】下，
为了【准确识别订单号、查询订单系统、并生成自然语言解释】，
我需要以下 3 类支持：

#### 1. 上下文信息

- 用户最近的聊天内容（`message_history`）
- 用户账号信息（`user_id`）
- 用户近期订单列表（`recent_orders`，包括订单号、商品名、创建时间）
- 用户可能没有提供订单号，但提供了模糊描述（如“我昨天买的手机”）
- 订单系统 API 配置（`order_api_config`）

#### 2. 能力支持

1. 意图识别能力：能识别用户是要：查订单 / 查物流 / 查售后 / 咨询其他问题
2. 槽位抽取能力：抽取订单号或根据用户描述推断订单（如“我昨天买的鞋”去最近订单列表匹配）
3. API 调用能力：调用订单查询 API；调用物流状态 API
4. 结构化决策能力：对订单状态进行解释：已发货 / 运输中 / 可能延迟 / 已签收
5. 自然语言生成能力：用不同语气回答：标准话术 / 安抚话术 / 给出下一步行动建议

#### 3. 工具清单（Tool List）

1. `search_order(order_id)`
   用途：查询订单基础信息
   输入：订单号
   输出：订单时间、价格、商品详情、状态（待发货 / 已发货）

2. `track_logistics(logistics_id)`
   用途：查询物流状态
   输入：物流单号
   输出：物流节点（揽收 / 运输中 / 派件 / 签收）

3. `find_possible_orders(user_id, product_hint)`
   用途：用户未提供订单号时，根据模糊描述匹配可能订单
   输入：用户 id + 商品关键词
   输出：可能的订单列表

#### 4. 输出格式要求（结构化输出）

```json
{
  "order_status": "已发货",
  "logistics_status": "运输中",
  "next_action": "预计2天内送达，如需加急可联系客服",
  "raw_data": {}
}
```

用户最终看到的是自然语言回复内容：

“您的订单已发货，目前正在运输途中，预计 2 天内送达。”

#### 5. 约束条件（Agent 的边界）

- 必须使用工具获取真实订单信息，不能凭空编造
- 当未找到订单时，要提示用户提供更多信息
- 多个订单匹配时，要主动询问：“您是指 xx 订单还是 xx 订单？”
- 输出必须符合 JSON Schema 结构化数据
- 不能泄露其他用户的订单信息
- 不可产生没有依据的物流时间预测

### 2. Agent 故事的技术实现方式（与上面 1-5 一一对应）

| Agent 故事中的需求 | 技术实现方式 |
| --- | --- |
| 上下文信息 | State 字段、消息历史、用户画像、订单列表 API |
| 工具清单 | LangChain Tools、MCP 工具、Function Calling |
| 能力支持 | LLM、提示词 |
| 输出格式 | JSON Schema、正则校验 |
| 约束条件 | 提示词约束、代码逻辑校验、重试机制 |

开发团队需要做的事情如下：

1. 理解需求：通过 Agent 故事明确：要识别订单号、要调用订单工具、要处理缺失槽位。
2. 准备上下文  
   在 State 中放入：
   - `recent_orders`
   - `user_id`
   - `message_history`
3. 配置工具：将 `search_order`、`track_logistics` 等工具绑定 Agent。
4. 模型选型：简单查单：小模型（快速）；复杂推断：GPT-4o、Claude Sonnet 等。
5. 设计提示词：包含：意图识别 / 槽位抽取 / 工具调用规则 / JSON 输出格式。
6. 实现校验：JSON Schema 校验输出 / 工具调用参数校验 / 兜底重试机制。

## 8.2.2 将用户旅程转为 Agent 工作流

### 1. 在传统产品设计中

“用户旅程（User Journey）”是产品经理必备的设计思维，它像一张交互地图，用于明确用户从哪一步进入产品、在哪些界面做出操作、最终如何完成目标。因为传统产品的操作流程是线性、前台可见、由用户驱动的，所以我们必须逐步定义每一步的操作路径和界面跳转。

### 2. 而在 AI 驱动的新一代产品中（如 Agent 应用）

产品流程不再完全依赖用户手动操作完成，而是转变为人机协作的工作流（Workflow）。其中大量操作由 AI Agent 自动完成，只有关键节点才需要用户输入或决策。

| 对比项 | 用户旅程（传统产品） | Agent 工作流（AGI 产品） |
| --- | --- | --- |
| 驱动主体 | 用户手动操作每一步 | Agent 自动执行为主，人类决策为辅 |
| 流程结构 | 线性路径、明确跳转 | DAG / 树形结构、动态路径 |
| 节点可见性 | 所有步骤均对用户可见 | 多数步骤封装在 Agent 内部，仅关键节点可见 |
| 设计关注点 | 界面跳转与交互一致性 | 判断交互点是否必要 + 是否信任 Agent |

### 输出【Agent 工作流】

传统 PRD：用户旅程图，描述用户一步步的操作。

AI 产品 PRD：用户只在发起 query 和关键节点做决策，Agent 需要理解需求、执行具体任务最终汇报结果。

示例：【订单查询 Agent 工作流】

1. 接收请求（Listen）  
   用户在前端输入问题：如：帮我查一下昨天买的鞋子什么时候到？  
   前端把用户 query、用户 ID、会话 ID 等发送到 AI 智能体。

2. 理解需求（Understand）  
   使用大模型进行意图识别：判断是否为【订单查询】而不是其他问题（如优惠券、商品咨询）。  
   进行槽位抽取：已提供订单号？只提供了时间 / 商品信息？（如“昨天”“鞋子”）

3. 补全信息（Collect Slots）  
   如果缺少关键槽位（订单号不明确）：调用“候选订单查询”工具获取该用户近期订单列表；  
   向用户追问：“是要查 Nike 跑鞋的订单，还是 Adidas 篮球鞋的订单？”

4. 选择工具并执行（Act）  
   确认订单后：调用订单系统 API：获取订单状态、物流单号；调用物流系统 API：获取最新物流节点。

5. 结果组装与决策（Reason）  
   将订单信息 + 物流信息进行结构化整理：当前状态（已发货 / 运输中 / 派件中 / 已签收...）；是否延迟、是否异常；是否需要提示用户进一步操作（如联系客服、申请售后）

6. 生成回复（Respond）  
   生成结构化结果（JSON 等），用于埋点和校验；再生成用户可读的自然语言回复：如：您的鞋子订单已发货，目前在运输中，预计 2 天内送达。

7. 记录与学习（Log & Learn，可选）  
   记录本次查询的输入输出，用于：监控效果（查询成功率、响应时长）；后续构建评测集和 Badcase 复盘。

### 工作流节点

- Listen：接收请求，获取 `user_id / session_id / query`
- Understand：意图识别、槽位提取
- Collect Slots：调用候选订单查询工具或向用户追问确认订单
- Act：调用订单系统 API、调用物流系统 API
- Reason：根据订单 + 物流数据进行判断，生成结构化结果
- Respond：生成自然语言回复
- Log & Learn：记录日志、埋点，用于后续评测和 Badcase 复盘

## 8.2.3 详细功能点会变为 prompt 设计

确定 workflow 之后，需要用提示词控制各个节点的输出，因此原有的功能详情会变为提示词内容。

### AI 产品 PRD 中提示词设计规范（Prompt Specification）

是一套结构化需求文档，它描述的是 Agent 应该如何“思考、推理、行动、输出”，提示词需要包括以下 5 部分：

1. 角色定位（Role Definition）  
   解释 Agent 扮演什么角色、具备哪些专业能力、在业务中承担什么任务。  
   PRD 需清晰写明：你是谁；你做什么；你对业务的理解范围。

2. 核心挑战（Key Challenges）  
   列出该任务中 LLM 可能出错的关键点。  
   示例：文本不规范、缺少关键信息；订单号缺失、描述模糊；工具调用需要参数完整；无法擅自编造订单信息。

3. 设计策略（Design Strategies）  
   常见策略包括：强化角色与约束、任务目标明确化、结构化指令列表（Checklist）、输出格式约束（JSON Schema）、异常情况处理策略、Few-shot 示例策略、工具绑定策略（Function Calling）。

4. 关键输出控制（Output Control）  
   控制输出符合预期：JSON Schema、字段解释、校验规则、错误处理逻辑、重试机制、不得编造内容。

5. 完整提示词（完整 System Prompt + Few-shots）  
   最终交付物。  
   在上述规范写完之后，PRD 最后才提供完整提示词文本，且必须是：可复制、可执行、可校验、可自动化测试。

## 8.2.4 原型图正在从 GUI 向 LUI 中的 GUI 演化

在传统产品中，原型图主要聚焦于 GUI（Graphical User Interface），即以图形界面呈现功能结构与交互流程。用户通过点击、滑动、选择等方式完成任务。但在基于大语言模型（LLM）的 AGI 应用中，用户界面逐步转向 LUI（Language User Interface）语言交互界面为主。

### 什么是 “GUI in LUI”？

在 AGI 驱动的系统中，大多数任务通过自然语言完成。界面不再是一个页面跳转的集合，而是对话流程中的“局部插入”。

当用户需要做出决策或操作时，系统会：

- 在对话窗口中动态插入可操作的 GUI 组件
- 以表单、卡片、图表等图形形式呈现内容
- 保留语言交互的连续性和上下文理解能力

示例：

- 用户输入：我的空调坏了，帮我报个修
- Agent 回复：请确认以下维修消息
- GUI 插槽：卡片式组件
  - 故障类型：空调无制冷
  - 时间可选：明天上午 10:00 / 明天下午 15:00
  - 地址确认：成都市天府五街 XX 号
  - 操作按钮：[修改地址] [确认提交]
- 用户点击：确认提交
- Agent 回复：已为您安排工程师张师傅上门，可在【个人中心 - 我的工单】中查看进度。

## 8.2.5【增加】数据评测集

以招聘 AI 助手为例：

| 数据类型 | 描述与用途 | 规模与要求 | 注意事项 |
| --- | --- | --- | --- |
| 招聘需求描述 | 用于测试需求识别、JD 生成及需求画像生成功能 | 50 份，做好关键信息和需求画像的人工标注 | 包含初级、中级、高级、专家等不同级别；覆盖技术、产品、市场、销售等不同职能；来源应多样化，避免单一模板；需求描述的详细程度不同 |
| （NER 模型微调） | 用于 NER 模型微调 | 5000 份，做好关键信息和需求画像的人工标注，也可基于人工标注 shot，借助大模型标注 | 同上 |
| 人岗匹配对 | 由专业招聘顾问手工标注的“优质匹配”案例，即一份 JD 和几份高度匹配的 CV，便于后期算法优化分析 | 30 组 | 覆盖技术、产品、市场、销售等不同职能；每组包含 1 份 JD 和 2-3 份被验证为“强匹配”或“合适”的简历；标注时应简单说明匹配理由 |

## 8.2.6【增加】评估体系

以招聘 AI 助手为例：

| 功能点 | 测试方法 | 测试指标 |
| --- | --- | --- |
| 招聘需求识别 | 测试人员（可借助大模型）将 AI 从口语化需求文本中提取的结构化信息，与人工标注的“黄金标准”进行比对，对准确率进行打分。 | 核心实体（职位名称、工作地点）F1 分数 > 90%；关键实体（经验年限、学历要求、核心技能）F1 分数 > 85% |
| 招聘需求补全 | 测试人员模拟完成从模糊需求到完整 JD 的全过程，结合对话质量与对话轮次进行主观打分。 | 任务完成率 > 95%；平均对话轮次 < 6 次 |
| 专业 JD 生成 | 将 AI 生成的 JD 与资深招聘顾问 / HR 编写的 JD 混合，由招聘 HR 盲评专业度、完整性和吸引力。 | 各项均分 > 4 分；不可分辨率 > 75% |
| 需求画像生成 | 测试人员（可借助大模型）对比 AI 生成的“需求画像”与手工定义的画像，检验其关键要素覆盖度和准确率。 | 平均关键要素覆盖度 > 95%；平均准确率 > 95% |
| 简历检索与匹配度分析 | 使用预先标注的黄金匹配对“数据集”，评估 AI 推荐排序列表的质量。 | NDCG@10 > 0.85 |

## 8.2.7【增加】产品功能设计及功能清单

对于直接利用大模型的功能点进行描述，如：

| 功能点 | 实现要点 | 关键技术 |
| --- | --- | --- |
| 招聘需求识别 | 支持用户输入口语化的招聘需求，系统通过 NLU 解析并结合知识库自动提取岗位、级别、核心技能等关键信息。 | NLU、NER 模型、向量模型 |
| 招聘需求补全 | 系统在已识别信息的基础上，利用“完整 JD 模板”自动发现并补充缺失内容；可通过上下文追问（如“该岗位的核心 KPI 是什么？”）以及多轮对话逐步完善需求。 | NLG、推荐算法 |
| 专业 JD 生成 | 将分散的信息整合成结构完整的 JD 初稿，描述专业、符合公司品牌调性。支持招聘 HR 在“转交审批 -> 审核 -> 决策”环节高效确认。 | NLG |
| 需求画像生成 | 将 JD 转化为多维度、带权重的“用户需求画像”，不仅包含显性要求，还可结合业务目标预测潜在需求（如转变方向、经验等）。 | NLG、向量化（含向量数据库） |
