# 实施计划模板

这份模板适合在 PRD 基础上继续往下生成实施计划。目标是把需求文档转成可推进的阶段，而不是把职能工作随便罗列出来。

## 计划标题

一句话说明这是哪个 PRD 的实施计划。

## 计划目标

- 这份计划要推动什么结果
- 计划完成后，团队应该进入什么状态

## 前置决策

列出在正式实施前已经确定，或必须先确定的关键决策，例如：

- 核心架构边界
- 数据模型
- 权限模型
- 平台形态
- 关键依赖

## 阶段拆分

按 3 到 6 个阶段拆分。每个阶段尽量是一个纵向切片，而不是纯技术职能切片。

每个阶段建议包含：

- 阶段名称
- 阶段目标
- 主要工作内容
- 涉及 AI 工具（本阶段主要用哪些 agent 或工具执行）
- 预期产出
- 验收标准（如何确认 AI 产出符合预期）
- 人工介入点（本阶段哪些节点需要人类决策或 review）
- 依赖和阻塞点

## Review 阶段（必选）

计划的最后一个阶段固定为 Review 阶段，在所有编码实施完成后执行。

- **阶段名称**：PRD 验收与回归测试
- **阶段目标**：对照 PRD 验证所有功能需求已正确实现，并确保既有功能未受影响
- **主要工作内容**：
  - 对照 PRD 功能清单逐项验收
  - 验证用户故事路径可走通
  - 验证 Agent 故事的输入输出与异常处理（如有）
  - 运行既有测试套件 + 新增专项测试
  - 覆盖 PRD 标注的边界条件和异常场景
  - 如有 AI 功能，使用 PRD 评测集评估
- **涉及 AI 工具**：Claude Code（自动化测试与验收）
- **预期产出**：验收报告（功能名称 | PRD 要求 | 实现状态 | 是否通过）、回归测试结果摘要
- **验收标准**：所有 PRD 功能项通过验收，所有既有测试通过，无 P0/P1 级缺陷
- **人工介入点**：最终验收确认、未通过项的优先级判定
- **依赖和阻塞点**：依赖所有前序实施阶段完成

## 建议顺序

说明为什么按这个顺序推进：

- 哪些高风险项被前置（通常是人工决策节点或技术验证）
- 哪些阶段依赖前一阶段的 AI 产出
- 哪些阶段可以并行（使用不同 agent 且输入不互相依赖）
- 哪些阶段之间需要人工 review 隔离（防止 AI 在错误产出上继续堆叠）

## 风险与依赖

- 哪些计划风险最需要提前管理
- 哪些外部依赖可能拖慢进度
- 哪些阶段需要额外评审或人工确认

## 下一步动作

说明计划落地后紧接着该做什么，例如：

- 进入 issue 拆分
- 启动技术验证
- 完成设计稿
- 拉齐跨团队依赖
