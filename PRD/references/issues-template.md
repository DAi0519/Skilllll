# Issue 拆分模板

这份模板适合在 PRD 和实施计划基础上继续拆成可执行任务。

任务面向 AI agent 执行，不是给人类分配工单。每个 issue 的核心产物是"一段可以直接交给 AI 执行的 prompt"，以及这个 prompt 应该路由到哪个 agent。

## 拆分原则

- 每个 issue 对应一个明确的 AI 执行单元
- 每个 issue 有清晰的输入、期望输出和验收方式
- 探索型问题（需要人决策）与实施型任务（AI 可直接推进）分开
- 依赖关系要显式写出，AI agent 不会自动感知上下文
- 同一个 agent 能连续执行的任务尽量合并，减少上下文切换
- 需要人类决策的节点必须显式标出，不能让 AI 自行猜测

## 可用执行主体参考

列举项目中实际可用的 AI agent 和工具：

- `Claude Code` — 本地代码生成、修改、重构、测试编写
- `Claude API` — 结构化数据处理、内容生成、分类打标
- `Cursor Agent` — IDE 内代码补全、跨文件修改
- `MCP: <工具名>` — 接入特定 MCP server 的能力（数据库查询、API 调用等）
- `人工决策` — 需要人类判断才能继续的节点，不能由 AI 代劳

## Issue 列表

每个 issue 使用下面结构：

---

### Issue 标题

一句话写清楚任务目标，从"能力"或"产出"出发，而不是"写代码做某事"。

#### 执行主体

指定负责这个 issue 的具体 AI agent，例如：

- `Claude Code`
- `Claude API`
- `MCP: postgres`
- `人工决策`

#### 执行提示

> 可直接粘贴给对应 agent 的 prompt，或足够清晰的指令描述。
>
> 应包含：任务目标、上下文说明、具体动作、输出格式要求。
>
> 避免模糊表述如"实现登录功能"，应写成"基于已有用户表结构，在 `src/auth/` 下实现手机号+短信验证码登录接口，返回 JWT token，包含单元测试"。

#### 输入

- 哪些文件、接口、数据或上游产出是这个 issue 的前提

#### 期望输出

- 这个 issue 完成后应该产出什么（代码文件、接口文档、数据结构、决策结论等）

#### 验收方式

- 如何确认 AI 的产出符合预期（运行测试、人工 review、接口联调等）

#### 依赖

- 依赖哪些其他 issue 的产出，或哪些人工决策节点

---

## Issue 排序建议

说明 issue 的推荐执行顺序：

- 哪些应该先做（通常是人工决策节点或技术验证）
- 哪些可以并行（输入互不依赖的 Claude Code 任务）
- 哪些必须等上游产出后才能开始
- 哪些 issue 之间需要人工 review 隔离（防止 AI 在错误基础上继续堆叠）
