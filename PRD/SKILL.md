# 中文 PRD

## Overview

把模糊想法整理成一份完整的中文需求文档。

这份文档面向 AI agent 执行，不是给人类开发者分配工单。PRD 定义产品意图，Plan 拆分交付阶段，Issues 把任务路由到具体的 AI agent（Claude Code、Claude API、MCP 工具等），并为每个 issue 提供可直接执行的 prompt。

每次执行这个 skill 时，都必须把产物落到用户目录下的 Markdown 文档里，不能只在对话里输出。

支持两种主要场景：

- 项目启动：用户给出一个新项目意图，希望得到完整项目需求文档
- 中途改动：用户在现有产品或系统上新增功能、调整流程或改造体验，希望得到功能级 PRD

并且支持一条后续链路：

- `PRD`
- `PRD -> Plan`
- `PRD -> Plan -> Issues`

先判断场景，再选模板。不要把这两类文档混成一种，也不要跳过 PRD 直接空拆任务。

## Mode Selection

### 1. 判断模式

优先根据用户输入判断是下面哪一种：

- `项目启动模式`
适用于“我要做一个……”“我想做个产品/工具/平台”“帮我定义这个项目”这类请求。通常还没有完整方案，重点是补全产品定义、目标用户、MVP、成功指标和风险。

- `功能改动模式`
适用于“在现有产品里加一个……”“优化某个流程”“新增某个模块/页面/能力”这类请求。重点是说清楚现状、问题、改动范围、用户故事、功能需求、实现决策、测试重点和非目标。

如果信息不足，就根据上下文做默认判断：

- 提到“新项目、从零开始、做一个产品、项目方向”时，默认 `项目启动模式`
- 提到“现有系统、当前流程、已有产品、改版、迭代、新增功能”时，默认 `功能改动模式`

## Workflow

### 2. 解释用户意图

无论哪种模式，都先把用户输入翻译成一个清晰的问题定义：

- 谁是目标用户
- 他们遇到了什么问题
- 这次想解决什么
- 预期结果是什么

如果信息很少，不要停在“信息不足”。先基于常识补出合理假设，再把假设显式写进文档。

### 3. 建立对应场景的上下文

如果是 `项目启动模式`，重点补全这些产品层信息：

- 目标用户是谁
- 他们的核心场景是什么
- 产品的核心价值是什么
- 第一个可交付版本应该包含什么
- 什么明确不做
- 如何判断项目是否有效

如果是 `功能改动模式`，重点确认这些现状信息：

- 现有产品或流程当前怎么工作
- 具体哪里有问题
- 哪类用户会受到影响
- 改动边界在哪
- 是否涉及兼容、迁移、权限、埋点或测试回归

如果工作区里有代码、文档、设计稿、Issue 或旧 PRD，优先吸收。项目启动模式下，缺少代码不应阻塞；功能改动模式下，缺少现状理解则要显式标记风险。

### 4. 用少量问题收敛关键决策

只追问真正会改变项目定义的问题。优先收敛：

- 模式判断是否正确
- 用户与场景
- 范围边界
- 成功标准
- 明显约束
- 高风险未决项

问题数量保持克制。用户如果只给一句意图，也允许直接先出完整初稿，再把未确认内容放到“假设”和“开放问题”。

### 5. 选择模板并输出文档

如果是 `项目启动模式`，使用 [references/prd-template.md](references/prd-template.md)。

文档规范以 [references/ai-product-output-standard.md](references/ai-product-output-standard.md) 为准。

最小交付集包含 9 个核心部分，缺一不可：

**1. 需求背景与目标**
问题描述、目标用户、业务目标、量化成功指标（必须可测量）。

**2. 现有流程与用户故事**
当前用户路径（逐步描述）、流程痛点、AI 替代/增强的具体节点。
用户故事格式：`作为【用户角色】，在【某场景】下，为了【目标】，需要【能力/支持】`

**3. Agent 故事**
每个 Agent 故事必须包含以下 5 个子部分：
- **上下文信息**：列出所有需注入的 State 字段（字段名 + 来源 + 说明）
- **能力支持**：列出 Agent 执行该任务所需的推理/调用能力（编号列表）
- **工具清单**：表格格式 `工具名 | 用途 | 输入 | 输出`
- **输出格式要求**：结构化 JSON Schema（会话开始建议 + 会话结束结果两层）+ 自然语言输出规格
- **约束条件**：Agent 的边界规则，必须写清降级路径和兜底机制（不能写"模型会处理"）

Agent 故事格式：`作为【Agent 角色】，在【任务场景】下，为了【目标】，需要【上下文/工具/能力支持】`

**4. 用户旅程与 Agent 工作流**
必须同时包含：
- 用户视角旅程（用户看得见的步骤）
- Agent 视角工作流表格（阶段 | 说明 | 用户可见？）
- **Mermaid flowchart**，节点必须覆盖：Listen → Understand → Collect Slots → Act → Reason → Respond → Log & Learn

**5. 界面草图与交互说明**
AI 产品界面以 **LUI（语言交互）为主，GUI 组件为辅**：
- LUI 主线：用户通过自然语言发起操作的对话流
- GUI 插槽：在对话流的关键决策节点插入可操作的卡片/表单（不是独立页面跳转）
- 关键确认节点：哪些操作必须用户显式确认，不能由 Agent 自行决定
- 用 ASCII 草图说明主要界面结构

**6. 功能设计与功能清单**
使用表格格式：`功能名称 | 目标 | 实现要点 | 关键技术 | 输入 | 输出`
每行对应一个功能模块，直接利用大模型的功能节点必须出现在此表中。

**7. Prompt 设计规范**
必须包含以下 5 个编号子部分：
- **7.1 角色定位（Role Definition）**：Agent 扮演什么角色、具备哪些专业能力、业务边界是什么
- **7.2 核心挑战（Key Challenges）**：列出该任务中 LLM 最容易出错的关键点
- **7.3 设计策略（Design Strategies）**：角色约束、结构化输出约束、Few-shot 策略、异常处理策略、工具绑定策略
- **7.4 关键输出控制（Output Control）**：JSON Schema、字段说明、校验规则、错误处理逻辑、重试机制
- **7.5 完整提示词**：可复制、可执行的完整 System Prompt + Few-shot 示例，这是最终交付物，不能省略

**8. 数据评测集设计**
使用表格格式：`数据类型 | 描述与用途 | 规模要求 | 标注方式 | 覆盖范围 | 注意事项`
必须包含：正向样本、边界样本、Badcase 集（持续积累）。

**9. 评估体系**
使用表格格式：`功能 | 测试方法 | 核心指标 | 达标阈值 | 上线门槛`
每个 AI 功能模块必须有对应行；达标阈值必须是数字，不能写"较高""合理"。

---

加上固定部分：非功能性需求（性能/安全/埋点/监控/灰度/降级/合规）、非目标、约束与依赖、风险、假设、开放问题、后续阶段建议

**当产品以 Agent 为核心时**，还必须额外输出 Agent 专属交付物（追加在文档末尾）：

- **A. Agent 职责说明**：名称、角色定位、核心能力列表、典型异常处理（每种异常对应兜底行为）
- **B. Inputs 定义**：表格 `字段 | 来源 | 说明`，覆盖用户输入、系统字段、历史数据
- **C. Outputs 定义**：结构化输出（供系统校验/埋点）+ 自然语言输出（供用户展示），两层分开描述
- **D. Agent 数据流时序图**：**Mermaid sequenceDiagram**，覆盖 用户→前端→Orchestrator→Agent→LLM→工具→数据库 的完整流转

如果是 `功能改动模式`，使用 [references/feature-prd-template.md](references/feature-prd-template.md)。

默认包含这些部分：

- 背景与问题
- 改动目标
- 当前现状
- 方案概述
- 用户故事
- 功能需求
- 边界与异常场景
- 实现决策
- 测试决策
- 非目标
- 风险与依赖
- 假设
- 开放问题

写作要求：

- 用中文写
- 输出要像一份独立文档，而不是聊天纪要
- 需求描述优先从用户行为、产品能力和业务结果出发
- 如果某些信息是推断出来的，要显式标记为“假设”
- 项目启动模式下，先把产品定义说完整，再谈实现方向
- 功能改动模式下，先把现状和改动边界说清楚，再谈方案细节
- 不要把文档写成工单拆解或代码设计稿，除非用户明确要求

#### 5.1 落盘输出

无论本次产出层级是 `PRD`、`PRD -> Plan` 还是 `PRD -> Plan -> Issues`，都必须执行落盘。

先判断这次是“新建”还是“更新”：

- 如果用户是在让你首次生成一份 PRD、计划或 issue 拆分，创建新文档
- 如果用户明确要求“更新对应文档”“修改刚才那份”“在原文档上调整”“继续补充当前文档”，就在当前输出文档上直接更新
- “当前输出文档”优先指本轮上下文里已经明确的文件路径；如果用户给了明确路径，以用户提供路径为准
- 如果用户没有给路径，但当前会话里这个 skill 上一次已经产出过文件，默认更新那一份对应文档
- 如果用户要求更新，但没有可识别的对应文档路径，就先明确说明无法定位旧文档，再按当前需求新建一份文档

更严格地执行下面规则：

- 只要用户是在同一个需求上继续补充、改写、收敛、扩展、拆 plan、拆 issues、补验收标准、补开放问题，默认都视为“更新当前文档”
- `PRD -> Plan` 和 `PRD -> Plan -> Issues` 的继续下游动作，默认也更新当前文档，把新章节追加到同一份文档中，而不是自动新建文件
- 只有在用户明确要求“另存一份”“新开文档”“保留旧版同时生成新版”“导出独立计划文档”时，才允许新建第二份文件
- 如果用户一边说“更新当前文档”，一边又要求“保留原文不动”，优先理解为需要另存新文件，并在回复里说明原因
- 不要因为产出层级变化就自动拆成多份文件；默认一条需求链路维护一份主文档

在”新建文档”时执行下面动作：

- 在**当前工作目录**下的 `PRD-Outputs/` 子目录中创建 Markdown 文档；如果目录不存在，先创建目录
- 创建 `PRD-Outputs/` 目录时，必须同时在该目录下创建 `.gitignore` 文件，内容为 `*`（忽略目录内所有文件），确保 PRD 文档不会被意外提交到版本控制，保护需求文档的隐私性。如果 `.gitignore` 已存在则跳过
- 当前工作目录是指 Claude 当前运行所在的项目根目录，不是用户的 `$HOME`
- 文件名使用时间戳加主题短名，避免覆盖旧文档
- 主题短名优先从需求主题、项目名或功能名提取，转成简短可读的 slug
- 如果主题不明确，使用 `untitled`

推荐文件名规则：

- 仅 PRD：`YYYY-MM-DD-HHMM-主题-prd.md`
- PRD + Plan：`YYYY-MM-DD-HHMM-主题-prd-plan.md`
- PRD + Plan + Issues：`YYYY-MM-DD-HHMM-主题-prd-plan-issues.md`

落盘要求：

- 文件内容应包含完整产物，而不是聊天摘要
- 对话里可以给精简版说明，但必须同时告知用户文档已保存，并附上完整文件路径
- 后续继续拆 plan 或 issues 时，默认更新当前主文档，不额外新建重复文件
- 只有在用户明确要求拆成独立文件时，才为 plan 或 issues 新建单独文档
- 如果是在修改已有产物，则直接更新对应文档，不额外新建重复文件
- 更新已有文档时，应先读取原文档，再在原内容基础上修改，而不是重写一份脱离上下文的新草稿

### 6. 根据需要继续下游链路

只有用户明确要求时，才在 PRD 之后继续生成计划和 issue 拆分。不要自行判断"当前任务明显需要继续"，默认止步于 PRD。

#### 6.1 生成实施计划

使用 [references/plan-template.md](references/plan-template.md)。

计划的目标不是罗列部门分工，而是把 PRD 转成几个连续的交付阶段。

拆计划时遵守这些原则：

- 优先按可交付的纵向切片拆，而不是按前端、后端、测试横向拆
- 每个阶段都尽量能产生一个可验证的结果
- 先固化不容易变化的架构和边界，再安排具体功能落地
- 把高风险和高不确定性的内容前置
- 显式标出依赖关系、阻塞点和验收标准

每个阶段必须包含以下字段：

- **阶段名称与目标**
- **主要工作内容**
- **涉及 AI 工具**（本阶段用哪些 agent 或工具执行）
- **预期产出**
- **验收标准**
- **人工介入点**
- **依赖和阻塞点**
- **本阶段 Issues**（产出层级为 `PRD -> Plan -> Issues` 时必须填写，列出本阶段所有 issue 的编号和标题）

计划末尾附：

- 计划目标总结
- 前置决策列表
- 风险与依赖汇总
- 建议执行顺序（含可并行节点和人工 Review 隔离点）

#### 6.2 拆分 Issues

使用 [references/issues-template.md](references/issues-template.md)。

Issue 拆分的目标是把需求转成可以直接交给 AI agent 执行的任务单元，不是给人类开发者分配工单。

**Issues 必须嵌入对应的计划阶段中，不单独平铺为独立列表。**

每个阶段下列出属于该阶段的所有 issue，每个 issue 使用以下结构：

- **执行主体**：指定具体 AI agent 或标为 `人工决策`，不写”待定”
- **执行提示**：可直接粘贴给对应 agent 的 prompt，包含任务目标、上下文、具体动作、输出格式要求
- **输入**：前置文件、接口、数据或上游 issue 产出
- **期望输出**：完成后应产出什么（代码、文档、数据、决策结论）
- **验收方式**：如何确认 AI 产出符合预期
- **依赖**：依赖哪些其他 issue 或人工决策节点

拆 issue 时遵守这些原则：

- 每个 issue 必须指定具体执行主体（哪个 AI agent 或工具），不写”待定”
- 每个 issue 必须包含可直接用作 prompt 的执行指令，而不是模糊的任务描述
- 把探索型工作（需人决策）和实施型工作（AI 直接推进）分开，两类不要混在一个 issue 里
- 需要人工决策的节点要显式标为 `人工决策`，不能让 AI 自行猜测或跳过
- 依赖关系要显式写出——AI agent 不会自动感知上下文，必须在 prompt 或输入字段里补全所需信息
- 同一个 agent 连续可做的任务尽量合并，减少无谓的上下文切换

可用执行主体包括但不限于：

- `Claude Code` — 代码生成、修改、测试编写
- `Claude API` — 内容生成、结构化处理、分类打标
- `Cursor Agent` — IDE 内跨文件修改
- `MCP: <工具名>` — 特定 MCP server 能力
- `人工决策` — 需要人类判断才能继续的节点

#### 6.3 PRD 验收与回归测试（Review）

在所有编码实施完成后，必须增加一个 **Review 阶段**，对照 PRD 进行验收并做回归测试。这是计划中的最后一个阶段，不可省略。

**Review 阶段的目标：**

- 验证实现结果是否满足 PRD 中定义的所有功能需求和验收标准
- 确认非功能性需求（性能、安全、边界场景等）是否达标
- 执行回归测试，确保新功能没有破坏已有功能

**Review 阶段必须包含以下内容：**

1. **PRD 逐项验收**
   - 对照 PRD 中的功能清单，逐项确认每个功能是否已实现且符合预期
   - 对照用户故事，验证每条用户路径是否可走通
   - 对照 Agent 故事（如有），验证 Agent 的输入输出、工具调用、异常处理是否符合设计
   - 对照评估体系中的指标和阈值，验证是否达标

2. **回归测试**
   - 运行已有测试套件，确认所有既有测试通过
   - 针对本次新增或修改的功能，编写并执行专项测试
   - 重点覆盖 PRD 中标注的边界条件和异常场景
   - 如有 AI 功能，使用 PRD 中定义的评测集进行评估

3. **验收报告**
   - 输出验收结果清单：功能名称 | PRD 要求 | 实现状态 | 是否通过
   - 列出未通过项及原因
   - 列出回归测试结果摘要
   - 如有未达标项，给出修复建议和优先级

**在生成实施计划时，Review 必须作为最后一个阶段出现在计划中。** 其执行主体通常为 `Claude Code`（自动化测试与验收）+ `人工决策`（最终确认），具体可根据项目情况调整。

### 7. 做可执行性检查

完成后按 [references/ai-product-output-standard.md](references/ai-product-output-standard.md) 第 7 节自检清单逐项核查：

- 是否写清了问题、目标、用户和成功标准
- 是否同时包含用户故事和 Agent 故事
- 是否写出了 Agent 工作流（而不是只写页面流程）
- 是否包含 Mermaid 工作流图（项目启动）或时序图（Agent 专属交付物）
- 是否明确了上下文、工具、能力和输出要求
- 是否给出了 Prompt 设计规范（而不是只停留在功能描述）
- 是否定义了结构化输出与校验规则
- 是否写清了异常处理与边界条件
- 是否给出了评测集设计
- 是否给出了评估指标和达标阈值
- 是否说明了上线后的监控、灰度和持续优化方式

如果文档还不足以推动下一步，就继续补”假设””开放问题”或”待确认决策”。补完后仍输出完整文档给用户，并在文档末尾或对应字段处显式标注哪些内容不确定，不要静默补全后直接呈现为已确认结论。

一份合格的 AI 产品文档必须同时做到三点：面向人看的部分足够清楚（团队知道做什么），面向模型的部分足够具体（系统知道如何约束），面向评估的部分足够可执行（团队知道如何判断是否有效）。三点缺一都不算合格。

## Stop Conditions

当以下条件**全部满足**时，skill 执行完毕，停止工作：

1. **文档已落盘**：Markdown 文件已写入 `PRD-Outputs/` 目录（新建或更新），并在对话中返回了完整文件路径
2. **模式已标注**：文档开头已写明 `模式：项目启动` 或 `模式：功能改动`，以及 `产出层级`
3. **产出层级已交付**：
   - 若产出层级为 `PRD`，PRD 内容完整（通过第 7 节自检清单）
   - 若产出层级为 `PRD -> Plan`，PRD + 实施计划均完整
   - 若产出层级为 `PRD -> Plan -> Issues`，PRD + 计划 + Issues 均完整（含 Review 阶段）
4. **未确认内容已标记**：所有推断和假设已显式标注为"假设"或"开放问题"
5. **不再追问**：关键决策已收敛，或用户明确表示当前版本可以

当以下情况出现时，**不要停止**，继续工作：

- 用户要求"继续拆 plan"或"继续拆 issues" → 更新当前文档，继续下游链路
- 用户要求"修改/补充/调整" → 读取原文档，在原内容上更新
- 自检清单有未通过项 → 补完再交付

## Question Strategy

当输入模糊、利益相关方多、或方案分支多时，再读 [references/discovery-questions.md](references/discovery-questions.md)。

提问时遵守这些规则：

- 一次只问少量最关键的问题
- 先问会改变项目定位、范围和优先级的问题
- 尽量给用户容易选择的对比项
- 能从现有资料推断的，不要反问用户
- 如果用户只想快速得到一版文档，优先出完整草稿，再请用户修正

## Output Rules

默认输出 Markdown。

默认产物是一份完整 PRD，不是 GitHub issue。

默认同时包含两份输出：

- 一份保存在**当前项目目录下** `PRD-Outputs/` 子目录中的 Markdown 文档
- 一份在对话里的简要结果与文件路径说明

如果这次是更新已有文档，则默认行为是：

- 更新原文件
- 在对话里说明“已更新”而不是“已新建”
- 返回被更新文档的完整路径

如果用户是在同一条需求链路上继续推进，默认行为是：

- 继续沿用当前主文档
- 在原文档中补充新的 `Plan`、`Issues`、决策、假设或修订内容
- 除非用户明确要求拆分文件，否则不要创建额外文档

如果用户没有特别说明，默认只输出 `PRD`。

如果用户要求“顺手拆计划”或“继续拆任务”，就输出：

- `PRD -> Plan`
或
- `PRD -> Plan -> Issues`

开头先显式写明当前使用的模式：

- `模式：项目启动`
- `模式：功能改动`

如果继续下游链路，也显式写明：

- `产出层级：PRD`
- `产出层级：PRD -> Plan`
- `产出层级：PRD -> Plan -> Issues`

在最终回复里，始终补充：

- `保存路径：/absolute/path/to/file.md`

如果用户明确要求简版，可以缩短叙述，但保留这些骨架：

- 目标
- 用户与场景
- 范围
- 需求
- 非目标
- 假设
- 开放问题

如果用户只给一句意图，也要尽量生成完整结构，而不是只回问题清单。

## Guardrails

**文档结构：**
- 不要把”项目启动文档”写成功能 patch 说明
- 不要把”功能改动 PRD”写成宏大产品愿景
- 不要把未来很多阶段的想法都塞进当前范围
- 不要在 PRD 还模糊时过早细拆计划和 issue
- 不要省略非功能要求、风险和非目标
- 不要把推测写成事实

**AI 产品专属：**
- 不要跳过 Agent 故事，不能只写用户故事而不说明 Agent 需要什么
- 不要用”智能处理””自动完成””AI 优化一下”替代具体的 Agent 能力描述
- 不要省略 Mermaid 工作流图或时序图，文字描述不能替代可视化流程
- 不要把 Prompt 设计降级为”提示词思路”，必须给出可执行的完整提示词
- 不要省略评测集和评估体系，AI 产品没有评估等于没有验收标准
- 不要把”模型会处理”当成异常处理的完整答案，必须写清降级路径和兜底机制

**Issue 拆分：**
- 不要把计划拆成纯职能分工列表（前端组/后端组/测试组）
- 不要把 issue 写成给人类的工单，issue 必须面向 AI agent，包含可执行 prompt
- 不要在 issue 里写”执行主体：待定”，必须明确指定 agent 或标为”人工决策”
- 不要把 issue 写成含糊的大包任务，AI 无法从模糊描述推断出正确行为
