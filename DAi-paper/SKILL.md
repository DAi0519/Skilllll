# DAi-paper: 读论文

读论文不是做学术，是猎取思想。把别人的发现拆解成自己能用的认知。

这条 skill 参考了李继刚老师的 [ljg-skills](https://github.com/lijigang/ljg-skills) 写法做了优化，主要聚焦 `AIGC / 空间智能 / 交互` 方向论文。

## 格式约束

### Markdown 语法

- 输出用标准 Markdown，文件扩展名必须是 `.md`
- 标题从 `#` 开始，不跳级
- 加粗用 `**bold**`
- 默认不要 YAML frontmatter。Obsidian 里优先保证“打开就能读”，不是先看到一整块属性
- 论文元信息放在标题下面的简短信息区，不要把 `title / tags / identifier / source` 这些字段堆成属性面板

### ASCII Art

所有图表用纯 ASCII 字符。允许：`+ - | / \ > < v ^ * = ~ . : # [ ] ( ) _ , ; ! ' "` 和空格。禁止 Unicode 绘图符号。

### 模板权威性

输出结构依据 `references/template.md`。禁止参考旧笔记文件的章节结构——旧文件可能使用过期模板。

### 文件规范

- 时间戳：`date +%Y%m%dT%H%M%S`
- 可读时间：`date "+%Y-%m-%d %a %H:%M"`
- 文件名：`{时间戳}--paper-{简短标题}.md`
- 默认输出目录：`{当前项目根目录}/论文阅读/`
- 如果当前任务不在项目里，或拿不到项目根目录，再回退到 `~/Documents/notes/`

### Markdown 开头

```
# {论文标题}

> 作者：{作者列表}
> 
> 来源：{发表场所/年份} | 状态：{preprint / accepted / published} | 时间：{YYYY-MM-DD Day HH:MM}
> 
> 原文：{URL 或来源描述}
> 
> 配套：{代码仓库 / 项目页 / 无}
```

文件写入后报告路径。

## 红线（每条必须过）

1. *口语检验* — 你会这样跟朋友介绍一篇论文吗？不会→改。学术腔是默认敌人
2. *零术语* — 先用大白话落地，再顺带提术语名。如果必须用原文术语才能解释，说明还没懂
3. *短词优先* — 能用两个字说的不用四个字。「本文提出了一种新的框架」→「他们做了个东西」
4. *一句一事* — 每句只推一步
5. *具体* — 名词看得见，动词有力气。形容词能砍就砍
6. *开头给理由* — 问题部分的第一句让人想知道答案
7. *不填充* — 删学术套话（「近年来随着...的发展」「值得注意的是」）。每句干活
8. *信任读者* — 说一遍够了。不重复结论
9. *诚实* — 论文有硬伤就说有硬伤。看不懂的部分说看不懂

## 写作原则

四条核心原则，决定文章是"活人在说话"还是"机器在汇报"：

1. *一个锚点撑全文* — 找到一个具象的中心隐喻（一张图、一个场景、一个动作），让所有概念围绕它生长。不是并列罗列五个概念，是一根绳子串起来。锚点在「核心概念翻译」开头就要出现，后续章节可以反复回到它
2. *推理外显* — 模拟"一个人想明白的过程"，而非呈现"想明白之后的结果"。用"既然A是B，那能不能C也是D？"带读者一起推。让读者觉得结论差一步就是自己想到的
3. *变形替代定义* — 解释两个概念的关系时，把A连续变形成B，不要说"A和B是XX关系"。「把LSTM变形→看起来像ResNet」比「LSTM和ResNet是对偶的」有力十倍
4. *落点在能用* — 给出"这意味着你可以___"，而非"这让我们重新思考___"。读者读完要带走一个能动手的东西，不是一个值得沉思的感慨

## 工具箱（选用）

讲解论文时可以拿的工具，没有哪个是必须的：

- *类比* — 承重的，方法的关键组件都能映射上。沿着类比走一遍方法
- *ASCII 图* — 展示组件关系、数据流、结构对比。读者有概念脚手架后再画
- *餐巾纸速写* — 「以前这么想，现在应该这么想」的并排对比
- *好问题* — 把论文解决的困境变成一个让外行也好奇的问题
- *递进例子* — 从简单到复杂，一步步搭建理解
- *反问入链* — 遇到隐含假设，用问题打开

## 执行

采用这套主结构：

- `来源判断`
- `背景与问题`
- `理论基础`
- `核心概念翻译`
- `方法与结果`
- `审稿洞见`

这套结构是重新收口后的最终骨架，同时融合旧版里真正有价值的内容：

- `来源判断` 保留时间窗、论文状态、主来源、paper 类型判断
- `背景与问题` 负责讲清来龙去脉和具体卡点
- `理论基础` 负责补理解基础
- `核心概念翻译` 把原来的“翻译”和“核心概念”整合成一节
- `方法与结果` 合并作者的方法、作者做了什么、以及结果证据
- `审稿洞见` 合并洞见、博导审稿、启发

### 0. 先过时间闸

默认只读近 1 到 2 年的论文，以当前日期往回算 24 个月为硬边界，12 个月内优先。

- 用户只说“找篇论文 / 读几篇论文 / 推荐 paper”时，只选最近 24 个月内的论文
- 有多篇可选时，优先最近 12 个月，其次 24 个月内
- 用户给的是旧论文时，先明确告诉对方这篇论文超出默认时间窗；除非用户明确说要破例，否则不要直接开读
- 写最终 Markdown 时，要把论文年份、发表时间、是否在默认时间窗内写清楚

时间规则的目的不是装新，而是避免把已经被领域淘汰、替代、吸收的东西当成当前前沿。

### 1. 获取内容

- arxiv URL → WebFetch
- OpenReview URL → WebFetch
- 官方 proceedings / anthology / open access 页面 → WebFetch
- PDF → Read（注意 pages 参数限制）
- 本地文件 → Read
- 论文名称 → WebSearch

确保拿到：

- 标题
- 作者
- 摘要
- 发表或公开时间
- 研究背景
- 具体问题
- 理论基础或前置概念
- 核心方法
- 作者实际做了什么
- 结果

如果是 AI / 空间智能 / AIGC 方向，优先信源按这个顺序找：

- *预印本*：arXiv
- *审稿与争议*：OpenReview
- *正式收录*：NeurIPS / ICLR / PMLR / CVF Open Access / 3DV / ACL Anthology / CoRL 官方页面
- *补元数据*：DBLP / Semantic Scholar
- *配套材料*：项目页 / GitHub / 官方 demo 页

对这几个方向，优先识别这些 venue 和轨道：

- *通用 AI / 大模型 / AIGC*：NeurIPS, ICLR, ICML, ACL, EMNLP
- *视觉 / 3D / 生成*：CVPR, ICCV, ECCV, SIGGRAPH, 3DV
- *空间智能 / 具身 / 机器人学习*：CoRL, RSS, ICRA, IROS
- *交互 / HCI / 人机协作*：CHI, UIST, CSCW, IUI, HRI, TOCHI

先判断论文状态，再开讲：

- *preprint*：只有 arXiv / 预印本页，没有正式接收信息
- *accepted*：OpenReview 或官网明确写 accepted / spotlight / oral / poster
- *published*：正式 proceedings 或 journal 页面可确认

把这个判断写进最终 Markdown 的信息区和正文，不要把 preprint 当成已定论。

如果能找到代码、项目页、demo、数据集，额外记下：

- 代码仓库链接
- 项目页链接
- 是否开源模型 / 数据 / checkpoint

这些是配套材料，不是证据本身。论文讲论文，代码讲可复现性，别混。

### 1.5 定位这篇论文值不值得读

先用一句白话判断它是哪一类 paper：

- *新问题*：以前没人认真打这个点
- *新方法*：问题不新，但解法真的变了
- *新系统*：把已有东西拼成第一次能跑通的系统
- *新数据 / 新 benchmark*：主要价值在评测基座
- *工程堆料*：更大、更贵、更全，但认知增量有限

这句判断要写进正文。它能帮读者决定是细读、略读，还是只记结论。

### 2. 来源判断：先判它是什么论文，值不值得读

这一节必须先把论文身份说清楚，再进入内容。

- 是否在默认近 24 个月时间窗内
- 现在是 preprint、accepted 还是 published
- 哪些来源是主证据，哪些只是辅助材料
- 它更像新问题、新方法、新系统、新数据，还是工程堆料
- 值不值得细读，还是知道结论就够

### 3. 背景与问题：先交代背景，再落到具体问题

先讲背景，再讲问题，顺序不能反。

- *背景信息*：这篇论文属于什么方向，这个方向最近为什么值得看，行业或学界在卡什么
- *具体问题*：作者到底盯住了哪一个具体缺口、失败点、瓶颈或矛盾
- *旧路为什么不够*：之前的方法卡在哪，为什么不能继续靠堆数据、堆算力、堆工程补过去

不是「本文提出了一种新的 XXX 框架」，是「这类系统最近都卡在同一个地方，而作者盯住的就是这个卡点」。

### 4. 理论基础：像给初中生讲一样，把基础讲明白

把理解这篇论文必须先知道的理论基础讲清，但只讲承重的，不讲百科全书。

这一节默认按“讲给聪明的初中生听”来写：

- 先说这是什么，再说术语名，不能反过来
- 一句只讲一个点，不要半句定义半句推导半句历史
- 能用生活里的东西类比，就别先上抽象词
- 如果必须提术语，术语放后面括号里补，不要拿术语当解释本身
- 目标不是显得专业，而是让第一次接触的人也能跟上

优先整理 2 到 4 项：

- 关键定义：论文里最重要的对象、任务、设定是什么
- 前置理论：它借了哪些经典理论、常见假设、已有方法范式
- 关键机制：比如 attention、diffusion、RL、energy-based、Bayes、3D 表征、HCI 实验设计这些，在这篇论文里各自扮演什么角色
- 依赖关系：这些基础是怎么一步步托住作者方法的

目标不是炫知识，而是回答一句话：**如果你不知道这些基础，后面哪一步会听不懂。**

写法要求再加一条硬规则：

- 每个基础先用 1 到 2 句大白话解释，再决定要不要补术语名
- 如果一个初中生读完还会问“所以它到底是干嘛的”，就说明这段还得重写

### 5. 核心概念翻译：把原理翻成外行也能跟上的话

这一节整合原来的“翻译”和“核心概念”。目标不是分两遍讲，而是一次性把最关键的概念和它们之间的关系讲透。

*开头先立锚点*：找到一个具象的中心隐喻或画面，在这一节的第一段就亮出来。后面所有概念围绕这个锚点生长，不是并列罗列。

*推理带着读者走*：不要直接给结论。模拟"一步步想明白"的过程——"既然X是这样，那Y能不能也这样？"让读者觉得结论差一步就是自己想到的。

需要覆盖：
- 最关键的 1 到 3 个概念是什么
- 每个概念是什么，像什么，为什么重要
- 这些概念是怎么连起来托住整篇论文的

如果是空间智能 / AIGC 论文，额外优先回答：

- 它到底在“看空间”、还是“生空间”、还是“在空间里行动”？
- 它用的空间表征是什么？点、体素、网格、NeRF、Gaussian、token、latent world model，还是别的？
- 它的生成或决策发生在哪一层？像素层、特征层、3D 表征层、动作层？
- 它是真的有空间约束，还是只是看起来像有空间感？

如果是交互 / HCI / human-AI interaction 论文，额外优先回答：

- 人到底在系统里做什么？点、拖、说、示范、改、选、审，还是监督？
- 系统把人的输入当成什么？命令、偏好、约束、示例，还是纠错信号？
- 交互回路有多长？一次出结果，还是来回迭代？
- 论文优化的是哪种体验？速度、可控性、认知负担、信任、协作感，还是创作空间？
- 评测是真人在做任务，还是只拿离线指标替代？

这一节的子标题按内容需要组织，不必固定，但至少要覆盖“概念本身”和“概念之间怎么连”这两件事。

### 6. 方法与结果：别把方法和结果拆成两张皮

先讲作者的方法，再讲作者做了什么，最后讲结果如何支撑作者主张。三者要连成一条链。

至少交代清楚：

- 作者的方法是什么
- 关键步骤怎么串起来
- 作者具体搭了什么、改了什么、训练了什么、比较了什么、验证了什么
- 哪 2 到 3 个结果最说明问题
- 这些结果分别支持了作者的哪条主张
- baseline 是否合理
- 证据里有没有软肋、边界条件、失败案例、隐藏成本

如果论文数字很好看，但证据链不完整，要直接说出来。

### 7. 审稿洞见：把判断、洞见、启发收成一节

先提炼这篇论文真正值钱的洞见。

- 如果它真的有思想结晶，用一句话把它说出来
- 如果它只是工程改进，没有认知层面的新发现，也要直说
- 不要复述论文结论，要提炼你读完之后真正带走的判断

然后再进入审稿视角：

换身份：这个方向上带了二十年研究生的博导。学生拿着论文来找你，你判断这东西值不值得认真对待。

用白话说，像在办公室跟学生聊：

- *选题眼光*：问题值不值得做？真缺口还是人造缺口？
- *方法成熟度*：巧劲还是蛮力？有没有更自然的做法被忽略？
- *实验诚意*：baseline 公不公道？消融到位没？数字经不经得起追问？
- *写作功力*：最该说清楚的地方有没有偷懒？
- *判决*：strong accept / weak accept / borderline / weak reject / strong reject，一句话理由

如果是 AI 论文，额外追问四件事：

- *比较对象对不对*：有没有故意绕开最强 baseline
- *算力换分*：增益到底来自想法，还是来自更多数据 / 参数 / 采样 / 工程调参
- *评测是不是贴题*：benchmark 分数和论文声称解决的问题是不是一回事
- *演示欺骗性*：demo 很唬人，但失败案例、边界条件、成本有没有交代

如果是交互论文，再追问四件事：

- *用户研究真不真*：样本量、任务设计、对照组够不够支撑结论
- *实验生态位*：是在实验室里短时试玩，还是接近真实工作流
- *交互成本*：用户多花的时间、学习成本、心智负担有没有算进去
- *好看不等于好用*：demo 顺滑不代表长期可用，论文有没有碰长期使用的问题

好的说好，差的说差在哪儿。

落点在"能用"，不在"能想"。给出"这意味着你可以___"，而非"这让我们重新思考___"。

用三个视角试探连接，命中展开，没命中跳过，全没命中说「没有」：

- *迁移*：论文的某个机制/视角能移植升级我体系的某个零件吗？具体怎么接？
- *混搭*：论文的某个组件和我已有的东西组合能产生新东西吗？产出什么？
- *反转*：论文的做法和我的默认假设相反吗？该停下什么、开始什么？

### 8. 过红线

逐条扫红线。额外检查：

- 破公式——否定式排比全文不超过两处，三段式改两项或四项
- 变节奏——长短句交替
- 杀金句——听起来像可引用的，重写
- 查跳跃——逻辑每步可追

列修改清单确认后生成文件。

### 9. 生成 Markdown 文件

按文件规范获取时间戳，读 `references/template.md`，优先写入 `{当前项目根目录}/论文阅读/`。

如果目录不存在，先创建：

- `{当前项目根目录}/论文阅读/`

如果当前任务不在项目里，或项目根目录不可判定，再回退创建：

- `~/Documents/notes/`

生成前再做一次完整性检查：

- 在默认时间窗内；如果不在，正文已明确写出这是破例
- 论文状态已写明
- 主要来源已标明
- 背景信息已交代
- 具体问题已说清
- 理论基础已补齐
- 理论基础像讲给初中生一样直白，不靠术语硬撑
- 核心概念已讲清，并且互相连起来了
- 作者的方法已讲明白
- 作者做了什么已写成动作链，不是空结论
- 方法和结果已写成“主张-动作-证据”关系，不是只贴数字
- 审稿判断和启发已并进最后一节
- 需要双写时，Notion MCP / 连接器已被实际调用并回读核对成功
- 需要双写时，Notion 内容默认落在数据库条目里，而不是数据库外的普通页面
- 如果有代码 / 项目页，已记录
- preprint 和正式发表版本没混淆

### 9.5 默认双写：本地 + Notion

如果当前线程里已经有可写的 Notion 目标，**默认产出两份**：

- 本地 `.md` 一份
- Notion 数据库条目一份

这里的“已有可写的 Notion 目标”包括：

- 用户明确提到要输出到 Notion
- 用户给了 Notion 页面 / 数据库 URL
- 用户在当前线程里已经通过 Notion 连接器确认过目标库或目标页

只有在这两种情况下才不双写：

- 用户明确说只要本地文件，不要 Notion
- 当前线程里根本没有可确认、可写入的 Notion 目标

执行顺序固定：

1. 找论文并读完
2. 生成本地 Markdown
3. 用本地 Markdown 同步到 Notion
4. 回读 Notion 页面做核对

先完成本地 Markdown，再同步 Notion。**本地 `.md` 是正文真源，不是你临时写给用户看的摘要。**

这里的“同步到 Notion”不是嘴上说同步了，而是要真的调用 Notion MCP / Notion 连接器去创建或更新页面 / 数据库记录。

硬规则：

- 只要线程里已有可写的 Notion 目标，或用户明确要求同步到 Notion，就**必须实际调用** Notion MCP / Notion 连接器
- 默认目标是 **Notion 数据库条目**，不是数据库外面的普通子页面
- 只有用户明确说“只要普通页面，不要数据库条目”，才允许创建独立页面
- 不能只生成本地 Markdown 然后在回复里说“已同步 Notion”或“可同步到 Notion”
- 不能用普通文本假装完成 Notion 同步
- 如果当前环境没有 Notion MCP / 连接器可用，必须明确说明“因为工具不可用，本次没法完成 Notion 同步”，不能静默跳过
- Notion 同步没有实际执行成功，就不算完成双写

同步规则：

- 先生成并保存本地 `.md`，确认文件内容完整后，再写 Notion
- 在 Codex / Claude 这类带连接器的环境里，优先直接使用 Notion MCP / Notion 连接器完成创建、更新、回读核对
- 如果拿到的是数据库 URL，就在该数据库里创建或更新一条记录，不要在数据库外新建普通页面
- 如果拿到的是 Notion 页面 URL，要先检查这个页面里是否有目标数据库；能确认数据库时，优先写入数据库，而不是挂一个页面在下面
- 只有在线程里根本确认不到数据库、且用户又明确接受普通页面时，才退回创建普通页面
- Notion 正文默认来自本地 `.md` 的正文主体，不要另写一版“压缩版正文”顶替
- 如果 Notion 条目本身已经有标题属性，正文里不要重复写第一行 `# 标题`；从 Markdown 开头的信息区往下同步
- 数据库属性和正文分开处理：
  - 属性里放标题、链接、日期、状态、短摘要、类型等结构化字段
  - 正文里放完整笔记内容
- `Summary` 之类字段只能放短摘要，不能把它误当成正文同步完成
- 如果数据库的 `select` / `multi_select` / `people` 等字段校验失败，不要硬写错误格式
  - 先只写确定能通过的字段
  - 补不进去的作者、venue、标签，可以临时放进正文的元信息区
- 同步后要重新抓取 Notion 页面，确认两件事：
  - 属性写进去了
  - 正文确实是本地 Markdown 对应的内容，不是替代文本

如果用户没有特别说明，但线程里已经确认过 Notion 目标，也按上面的默认双写执行，不要只停在本地文件。

## 验收

- *时间过闸*：默认只读近 24 个月；破例时明确写出为什么破例
- *结构收口*：正文按 `来源判断 / 背景与问题 / 理论基础 / 核心概念翻译 / 方法与结果 / 审稿洞见` 这六段来写
- *问题勾人*：让不懂的人也想知道答案
- *背景够清*：读者知道这篇论文在什么大盘子里，为什么现在值得看
- *问题够实*：读者知道作者到底盯住了哪一个具体难点
- *基础讲清*：理论基础不是术语清单，而是后文理解所需的必要前提
- *像讲给初中生*：理论基础先用最简单明了的话讲清，再补术语
- *有锚点*：核心概念翻译部分有一个具象的中心隐喻，后续概念围绕它生长
- *带着推*：读者能感受到"一步步想明白"的过程，而非接收打包好的结论
- *概念翻透*：关键概念不只定义清楚，还讲清彼此关系
- *方法讲成动作*：不只说“提出了什么”，还说清“怎么做”
- *作者做了什么能复述*：别人听完能复述数据、流程、实验，而不是只记住一句标题党结论
- *结果不空转*：方法与结果部分说明“哪些动作带来哪些证据，哪些证据支撑哪些结论”
- *外行能跟*：不懂这个领域的聪明人读完能复述核心思路
- *博导像博导*：有判断力有分寸，最后一句判决
- *启发能动手*：审稿洞见部分的落点是"你可以___"，不是"值得思考___"
- *Notion 真同步*：需要双写时，必须真的通过 Notion MCP / 连接器写入并核对，而不是口头声明
- *默认进数据库*：需要 Notion 同步时，默认写进数据库条目，不在数据库外另挂一个普通页面
- *零割裂感*：读完像一个人在跟你说「我读了篇论文，发现了个有意思的事」
